## 進め方(Agentic Coding)

このリポジトリでは完全お任せ(vibe coding)ではなく、人間が判断を行うAgentic Codingで進める。

- 曖昧な点・仕様の解釈が分かれる点は勝手に判断せず、選択肢とpros/consを提示して確認する。
- 以下は実行前に必ずユーザーの確認を取る:
  - force push・`reset --hard`・ブランチやファイルの削除などの破壊的操作
  - アーキテクチャ・依存関係・データ形式に関わる大きな判断
  - PRのマージ
- ユーザーが行った変更を、明示的な依頼なしに巻き戻したり上書きしたりしない。
- 大きめの機能は、まず計画(SOW)を立てて合意してから実装に入る。計画は`docs/plans/`に保存する。
- セッション中に受けた重要な指示・繰り返し指摘された事項は、本ファイルに追記して蓄積する(追記時は`AGENTS_en.md`も同時に更新する)。

## 開発フロー

- GitHub Flowで進める。`main`への直接コミット・プッシュは禁止。
- ブランチ名は英語のkebab-caseで、`feature/xxx`・`fix/xxx`・`docs/xxx`・`chore/xxx`とする。
- TDD(t_wada流)で進める:
  1. Red: 失敗するテストを先に書く
  2. Green: テストを通す最小限の実装を書く
  3. Refactor: テストを通したままきれいにする
- コミットは小さく、1コミット=1論理変更。節目ごとにブランチへプッシュする。
- 不要な機能・抽象化を先回りで追加しない(YAGNI)。

## コミットメッセージ

Conventional Commitsに従う。

```text
<type>: <subject>   ← 英語・命令形・50文字以内

<body>              ← 日本語で変更の背景・内容を説明
```

- type: `feat` / `fix` / `docs` / `test` / `refactor` / `chore` / `style` / `ci`
- 例:

```text
feat: add todo filter feature

フィルター機能(すべて・未完了・完了)を追加した。
```

## PR・レビュー・マージ

1. フィーチャーブランチで作業し、PRを作成する(PR本文は日本語)。
2. PR本文に`\n`リテラルを書かない。実際の改行を使う(`--body-file`推奨)。
3. Copilotにレビューを依頼する(`gh pr edit <番号> --add-reviewer @copilot`)。
4. 指摘には修正コミットで対応し、各レビュースレッドに対応内容を返信してからresolveする。
5. 再レビューを依頼し、新規指摘ゼロ(No New Comment)になるまで4〜5を繰り返す。
6. CIの全ジョブがグリーンであることを確認する。
7. レビュー完了・CIグリーン・ユーザー承認の3つがそろってからマージする。

禁止事項:

- レビュー未完了・CI失敗状態でのマージ。「時間がかかっている」等の自己判断でスキップしない。
- 再レビュー目的でのPRのクローズ/再作成(履歴が壊れる。クローズ/リオープンでも再レビューはトリガーされない)。
- コミット・プッシュしただけで「完了」と報告すること。CI通過とマージ状態を確認するまで完了ではない。確認できない場合は「プッシュ済み、CIの確認待ち」と報告する。

## ドキュメント

### 整合性

- README・本ファイル・`docs/`の記載と、コードの実態(コマンド名・構成・依存関係)を常に一致させる。
- ファイルの追加・削除・リネーム時は、関連ドキュメントの記述も同じ変更の中で更新する。

### 日英同期

- 日本語のドキュメント`<名前>.md`が正、`<名前>_en.md`はその英語版。
- 日本語版を変更したら、同じコミットの中で必ず`<名前>_en.md`も更新する。
- `CLAUDE.md`や`.github/copilot-instructions.md`のような参照だけの薄いファイルには英語版を作らない。

### ADR(アーキテクチャ決定記録)

- 技術選択・設計判断をしたら`docs/adr/NNNN-title.md`に記録する(テンプレート: `docs/adr/0000-template.md`)。
- 既存ADRの上書き・修正は禁止。決定が変わったら旧ADRを非推奨化し、次の番号で新しいADRを作る(手順は`docs/adr/README.md`)。
- レビュー指摘を受けても技術的・環境的理由で現状を維持する場合は、該当コードに`Why:`コメント(1〜3行)を書き、ADRにも記録する。

## セキュリティ

- シークレット・トークン・認証情報をハードコードしない。例示では環境変数やシークレットマネージャを使う。
- 外部入力は検証し、ファイルパス・ユーザー入力をサニタイズする。
- 依存関係は安定版を使い、メンテナンスされていないライブラリを避ける。
- `.github/dependabot.yml`の`cooldown`設定はサプライチェーン攻撃対策の重要機能である。レビューツールに「無効なキー」等と指摘されても、公式ドキュメントで仕様を確認して修正で追従する。ユーザーの明示的な承認なしに削除・無効化しない。
- セキュリティに関わる設定(CI・依存管理・権限)の削除・緩和は、必ずユーザーに確認してから行う。
