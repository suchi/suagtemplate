# メンテナンスガイド — テンプレート自体の運用指針

このリポジトリ(テンプレート自体)をよりよい形に育てていくための指針と推奨手順。テンプレートから作った実運用リポジトリ側の手順は[setup-guide.md](setup-guide.md)。

## 原則

1. このリポジトリ自身もAGENTS.mdのフローに従う(フィーチャーブランチ+PR+Copilotレビュー+ユーザー承認でマージ)。
2. 1ルール1か所を守る([README.md](README.md)の設計方針参照)。新しい規範を足すときは正となる置き場所を1つ決め、他は参照にする。
3. 汎用のものだけをテンプレートに入れる。1プロジェクト限りの指示はsetup-guideのカスタマイズ候補に置く。
4. 変更したら必ず: 日英同期、[history.md](history.md)への追記、関連ドキュメントの整合。
5. 個別のアカウント情報・リポジトリ名をテンプレート・ドキュメントに書かない(過去の経験に基づくことを書くのはよい)。

## 変更の種類ごとの推奨手順

### 実プロジェクトからの知見の還元

1. 実運用リポジトリのAGENTS.mdに蓄積された追記を定期的に見返す。
2. 複数プロジェクトで有効だった指示だけを、汎用化した表現に書き直してテンプレートのAGENTS.md(または適切な正の置き場所)へ昇格する。
3. 何のための知見かをhistory.mdに記録する。

### フック・settings.jsonの変更

1. 変更後に`sh template-docs/tests/hook-tests.sh`を実行し、全ケースPASSを確認する。
2. ガードを追加・変更したらテストケースも同じ変更で追加する。
3. これらのファイルはprotect-config.shのask対象なので、確認に応じて進める(ガードを外す変更は特に慎重に)。

### コマンド・スキルの変更

1. 本体(英語)を変更し、`template-docs/ja/`の日本語訳を同期する。
2. 手順番号やコマンド名を変えたら、参照箇所(vibe-coding.md等)をgrepで確認する。

### 公式仕様への追従(定期)

- 四半期に一度程度、[references.md](references.md)の各文書を見直し、仕様変更(設定ファイル形式・AGENTS.mdサポート状況・セキュリティ推奨)に追従する。確認したら参照日を更新する。
- 新しいエージェント/ツールに対応するときは、AGENTS.mdの「エージェント別の設定ファイル」表・agent-notes.md・references.md・setup-guide.mdをそろえて更新する。

## リリース運用

- 大きな構成変更の節目でタグを打つ(例: `v1.0.0`)。
- テンプレートは複製されて使われるため、変更は既存の実運用リポジトリへ自動反映されない。必要な変更は各リポジトリが手動で取り込む。破壊的変更(ファイル名・フック挙動の変更)はhistory.mdに明記する。

## PRに出す前のチェックリスト

- [ ] フック変更時: `sh template-docs/tests/hook-tests.sh`が全PASS
- [ ] 日英同期(変更した日本語mdの`_en.md`、英語ベースのコマンド/スキルの`ja/`訳)
- [ ] `grep -rn "TODO(project)"`の結果が意図どおり(テンプレートのプレースホルダが消えていない)
- [ ] アカウント情報・個別リポジトリ名が入っていない
- [ ] history.md(+_en)に追記済み
