# メンテナンスガイド — このリポジトリ自体の運用指針

このリポジトリ(テンプレート集)をよりよい形に育てていくための指針と推奨手順。テンプレートから作った実運用リポジトリ側の手順は[setup-guide.md](setup-guide.md)。

## 原則

1. このリポジトリ自身もトップの`AGENTS.md`のフローに従う(フィーチャーブランチ+PR+Copilotレビュー+ユーザー承認でマージ)。
2. テンプレート間同期: `template/`と`template_ja/`は同じ構成・同じ意味内容を保つ。片方を変更したら同じ変更の中でもう片方にも反映する(詳細はトップの`AGENTS.md`)。
3. 1ルール1か所: 規範の正となる置き場所を増やさない。新しい規範を足すときは正の置き場所を1つ決め、他は参照にする。
4. 汎用のものだけをテンプレートに入れる。1プロジェクト限りの指示はsetup-guideのカスタマイズ候補に置く。
5. 変更したら必ず: 両テンプレートへの反映、[history.md](history.md)への追記、関連メタ文書の整合。
6. 個別のアカウント情報・リポジトリ名・件数などの詳細をテンプレート・ドキュメントに書かない(過去の経験に基づくことを書くのはよい)。
7. メタ文書(トップREADME・AGENTS.md・docs/配下)は日本語で書く。英語の成果物は`template/`そのものが担う。

## 変更の種類ごとの推奨手順

### 実プロジェクトからの知見の還元

1. 実運用リポジトリのAGENTS.mdに蓄積された追記を定期的に見返す。
2. 複数プロジェクトで有効だった指示だけを、汎用化した表現に書き直して両テンプレートのAGENTS.md(または適切な正の置き場所)へ昇格する。
3. 何のための知見かをhistory.mdに記録する。

### フック・settings.json等のコード類の変更

1. 両テンプレートで同一内容にする(バイト単位で一致。テストが同一性を検証する)。
2. 変更後に`sh docs/tests/hook-tests.sh`を実行し、全ケースPASSを確認する。
3. ガードを追加・変更したらテストケースも同じ変更で追加する。
4. これらのファイルはprotect-config.shのask対象なので、確認に応じて進める(ガードを外す変更は特に慎重に)。

### コマンド・スキル・文書類の変更

1. 対訳として両テンプレートに同じ変更を反映する(英語版と日本語版で意味を一致させる)。
2. 手順番号やコマンド名を変えたら、参照箇所(vibe-coding.md等)をgrepで確認する。

### 公式仕様への追従(定期)

- 四半期に一度程度、[references.md](references.md)の各文書を見直し、仕様変更(設定ファイル形式・AGENTS.mdサポート状況・セキュリティ推奨)に追従する。確認したら参照日を更新する。
- 新しいエージェント/ツールに対応するときは、両テンプレートのAGENTS.mdの「エージェント別の設定ファイル」表・agent-notes.md・references.md・setup-guide.mdをそろえて更新する。

## リリース運用

- 大きな構成変更の節目でタグを打つ(例: `v1.0.0`)。
- テンプレートは複製されて使われるため、変更は既存の実運用リポジトリへ自動反映されない。必要な変更は各リポジトリが手動で取り込む。破壊的変更(ファイル名・フック挙動の変更)はhistory.mdに明記する。

## PRに出す前のチェックリスト

- [ ] 両テンプレートに同じ変更が反映されている(コード類は`docs/tests/hook-tests.sh`の同一性チェックがPASS)
- [ ] フック変更時: `sh docs/tests/hook-tests.sh`が全PASS
- [ ] `grep -rn "TODO(project)" template template_ja`の結果が意図どおり(プレースホルダが消えていない)
- [ ] アカウント情報・個別リポジトリ名が入っていない
- [ ] history.mdに追記済み
